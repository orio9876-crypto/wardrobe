# Technical plan — V1

## Evidence, recommendation, assumptions
**Confirmed facts.** The client is React 19 + Vite 6 JavaScript with plain CSS, a minimal manifest/service worker, `@unpic/react`, Phosphor icons, Instrument Sans, Sharp, and two Vite `apply: "serve"` plugins. `scripts/import-job-api.mjs` calls OpenAI server-side, writes `data/library.json`/filesystem jobs, and has crop → garment → modeled review states. There is no deployable server, database, auth, tenancy, queue, tests, router, or outfit application. Existing browser edits/deletes are localStorage overlays; all visitors share the local data directory. See `PROJECT_ANALYSIS.md` for the verified contract.

**Recommendation (V1).** Retain React’s presentation strengths but replace the Vite middleware/local filesystem boundary with a TypeScript full-stack web application: React UI plus a server/API, PostgreSQL, private object storage, a durable queue/worker, and managed identity. This is the smallest architecture that makes user isolation, deletion, long-running AI work, and auditability real. Alternative: keep Vite plus custom Express/filesystem—lower initial rewrite cost but poor deployment/security/queue semantics. Alternative BaaS-only functions—fast but background image jobs and vendor portability become awkward. Tradeoff: TypeScript/schema/managed-service setup first, but greatly lower data-loss and authorization risk. Provider selection is an implementation decision after pricing, Israel-region/DPAs, OAuth, and team familiarity validation; do not select a vendor purely in these documents.

**Explicit assumptions.** PostgreSQL supports row-level access through application transactions (and optionally RLS as defense in depth), object storage supports private buckets/signed URLs, queue supports retries/idempotency/dead-letter handling, and deployed server/worker share secrets only. Confirm actual AI model/API capability at implementation time; model names in current local code are not an endorsement or a V1 contract.

## Architecture and stack
* **Web:** React, TypeScript, modern server-rendering-capable framework or Vite client plus TypeScript API; choose the former if it gives coherent auth/session/route boundaries. Reuse CSS concepts, not its English/local assumptions. Use a mature i18n library with locale routing/root `dir`, `Intl`, logical CSS, Hebrew-capable licensed fonts, and a route-level error boundary.
* **API:** authenticated JSON/multipart initiation endpoints; direct-to-private-storage upload via short-lived single-object signed POST/PUT; server verifies completion before enqueue. API keys never enter browser bundles.
* **Data:** managed PostgreSQL + migrations in implementation (not now), ORM/query layer with generated schema types, transaction boundaries, encryption-at-rest provider settings, daily backups and restore test.
* **Assets:** private object storage prefixes such as `users/{userId}/sources/{id}/original`, `derived`, and `generated`; database stores object keys, not permanent URLs. Issue short-lived, authorization-checked viewing URLs. Originals and generated outputs never share a public CDN path.
* **Jobs:** managed queue plus worker container/function. States live in PostgreSQL; worker leases jobs, uses idempotency keys, records attempts/provider request IDs/cost, and emits status through polling first (SSE later). One import batch active per user in beta.
* **PWA:** retain manifest concept but build an authenticated shell; cache static assets only initially. Avoid service-worker caching private API/image responses unless encrypted/user-scoped strategy is validated. Installability is V1-ready; offline mutation queue is later.

## AuthN, AuthZ, security, privacy
Use OAuth/OIDC authorization code flow with PKCE for Google and Apple through a managed auth service or audited library; map each external identity to one `users` row. Design identity table for email/password later, but do not collect passwords V1. Reauthenticate for account deletion. Store secure httpOnly/SameSite session cookies (or provider’s equivalent), protect state-changing requests against CSRF, rotate sessions, and never trust client `userId`.

Every repository query/mutation receives server-derived actor identity and constrains `owner_user_id`; storage signing validates ownership and allowed object purpose; queue payload contains only record IDs. Add DB RLS where supported, but do not call it a substitute for application checks. Audit access/changes/deletion requests. Rate-limit identity, upload, recommendation, regeneration, and signed-url paths by user/IP; enforce daily AI quotas, batch concurrency, maximum retries, content limits, request IDs, structured security logs, and secrets manager/key rotation. Validate magic bytes/decode with bounded pixel count, strip EXIF except needed orientation/time policy, reject SVG/unsafe formats for V1, virus/malware scan or managed scanning hook before worker access, and store user-provided text safely.

Privacy design: explicit purpose-specific consent records; no training/use beyond providing requested processing absent separate opt-in; no sharing/public publication; minimised worker payloads; deletion and retention scheduler; encrypted transport/storage; support access/export request workflow later. Israeli users require a privacy/legal review of controller/processor, cross-border transfers, data subject process, and notice language; this is not legal advice.

### Image classes and retention
| Class | purpose / retained assets | consent & access / deletion |
|---|---|---|
| Wardrobe source | evidence for detection; original plus approved candidate crops while source exists | processing consent; owner/worker only; delete source deletes derived candidates unless independently retained approved item is confirmed |
| Daily outfit | wear/history evidence and detection source; original/crops separately flagged | explicit daily-outfit action; owner/worker only; deleting image preserves wear event only if user selects that option |
| Body reference | optional future visualization only; original and derived representation separate | separate explicit consent; never needed for core recommendations; delete invalidates derived visualization assets |
| Generated visualization | optional output, never wardrobe source of truth | explicit per-feature consent; owner only; delete output and provider references; beta disabled by default |

Set exact retention values only after legal/product decision. V1 recommended defaults: retain originals/crops only while their user-visible source or approved item needs provenance; purge rejected candidates/temporary provider artifacts within 30 days (earlier where possible); failed job artifacts within 7 days; deletion jobs continue until verified purge, with backup-expiry disclosed.

## AI and image pipeline
1. Browser selects/captures explicit images and asks for purpose; client creates a resized preview/upload derivative (e.g., bounded dimension) but original policy is explicit.
2. API creates source-image/import-batch/detection-job rows, validates upload completion, calculates content hash/perceptual fingerprint, and queues detection.
3. Worker normalizes orientation/color safely, produces bounded processing derivative, caches by source hash + pipeline version, uses an **image-understanding model/API category** with structured schema to identify up to a beta cap of garments, boxes/masks, metadata and field confidence. It stores raw provider response separately/restricted and normalized candidates.
4. For each candidate, worker creates a crop/mask. **Segmentation/extraction** is a specialist segmentation/masking API/model or deterministic image process where quality is sufficient; generative reconstruction is only an opt-in later/beta refinement because it can invent details. Preserve source crop as evidence and label generated visual assets.
5. Low-confidence crop/category/color, conflicts, extraction failures, or possible duplicates enter review. Only an explicit user approval transaction creates an eligible wardrobe group/instance.

Use model categories, not hard-coded model names: multimodal structured vision for detection/metadata; segmentation model/service for masks; embedding model suitable for image retrieval; multimodal ranker restricted to candidate JSON/images; language model for explanation; image-generation/virtual-try-on only separately consented and never core. Verify current availability, data handling, pricing, regional processing, schema support and eval quality before vendor choice. Current detection schema/prompt normalization and source-faithful garment prompt are reusable **reference material**, after threat/privacy review; the current modeled-identity workflow must not be ported into core V1.

## Metadata, duplicates, and recommendation
**Taxonomy.** Versioned controlled enums with user-visible localized labels: category/subcategory; primary/secondary colors and palette; pattern; material; fit/silhouette/length; styles; seasons; warmth 1–5; formality 1–5; visual prominence; modesty attributes; brand/size (nullable); condition; availability; laundry state optional; source; AI field confidence; user-confirmed fields. Preserve unknown rather than fabricate; store free notes separately.

**Duplicate retrieval, not automatic merging.** Create embedding from candidate crop (and optionally source context), perceptual hash, color vector, categorical/pattern/detail features, source-image lineage, and user-confirmed link history. Retrieve same-user approved group/instances with an indexed vector search; score an explainable weighted combination and apply threshold bands: below threshold no prompt; middle/high prompt user; never auto-merge. “Same physical item” creates a sighting link; “identical additional” creates another instance in the group; “similar different” creates another group; decisions train only per-user rules/negative pairs. Store model/pipeline version. This costs an embedding call/storage but prevents the false certainty of image similarity.

**Hybrid outfit engine.**
* A deterministic generator filters `approved && active && available && condition!=damaged && (laundry!=dirty)` and honors preference category scope, required/excluded IDs, recency, weather, occasion/formality, season/warmth, modesty, color/pattern/silhouette compatibility. Compose bounded templates (e.g., top+bottom, dress, optional layer/shoes/accessory), enumerate with diversity caps, and return why candidates were excluded.
* An AI ranker receives only the valid candidate item IDs and compact verified metadata/thumbnail references; it ranks, scores harmony/context/weather confidence, flags weaknesses, and returns schema-validated explanations. It cannot return item IDs outside supplied candidates. Start with a deterministic fallback ordering on error/quota.
* Personalization updates transparent per-user weights/signals from favorites, wear events, replacements and explicit feedback; keep general fashion, personal, context, weather, and confidence scores independently. No custom fashion model V1; consented aggregate model research is future.

Evaluate offline with a manually labeled Israeli-context test set (validity, diversity, explanation faithfulness, Hebrew quality) and online via save/wear/negative feedback, rejection rate, duplicate correction rate, cost, latency, and safety audits. Never interpret engagement alone as fashion truth.

## Cost, operations, deployment, testing
Main cost drivers: high-resolution upload storage/egress, vision calls per image, segmentation/generative extraction, embeddings, ranking calls, and optional visualization. Beta controls: 10 selected photos/batch (validate), one active batch/user, resize processing input, hash/cache results, no reanalysis without a material change, prefilter before AI, bounded candidates/ranker calls, daily/monthly quotas, retry ceiling, cancellation, provider timeout/backoff, and per-user/provider cost ledger. Disable visualization and full video processing in beta.

Environments: local (fake providers/object storage emulator; no production data), staging (isolated OAuth callbacks/bucket/database), production (separate project/secrets/bucket). CI: lint/typecheck/unit tests, component RTL/LTR tests, integration tests against ephemeral DB/storage/queue fakes, security authorization tests, visual RTL screenshot tests, upload corruption tests, worker idempotency/retry/deletion tests, and accessibility checks. E2E covers Google/Apple test identities where possible. Observability: structured redacted logs, trace/job/source IDs, metrics/dashboards for queue age, success/failure, cost/user, storage, duplicate decisions, recommendation validity, signed-url errors, alerts and provider circuit breaker.

Deploy web/API and worker separately with health checks, least-privilege service identities, controlled rollout and migration rollback plan. Backups and restore test are V1 operational requirements.

## Main risks
1. **Product:** users may not trust imperfect detections or see value before enough approved items. Mitigate by fast manual add, review transparency, sparse-wardrobe recommendations, and private beta interviews.
2. **Technical:** garment identity/duplicate accuracy from photos is intrinsically uncertain. Mitigate with candidate-only state, source evidence, human resolution, reversible decisions, and measured thresholds—not automatic merge.
3. Provider quality/cost, OAuth/App Store policy, photo permissions, Hebrew RTL truncation, deletion across backups/providers, and one-developer operational load require validation gates.
